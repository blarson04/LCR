# Multifamily Screener — v2.1 Build Spec: The Nowcast Layer

*Companion to `decision-log.md` (why), `v1-build-spec.md` (v1 how), `v2-plan.md` + `v2-findings.md`
(the rigor pass). Read all before building.*

*Context for the build assistant: the author is newer to coding but has solid real estate
fundamentals. Explain modules as you build, prefer clear code over clever code, and build in
small runnable increments. The v2 model (de-duplicated 8-indicator scheme, model 2.0.0) is the
frozen foundation — v2.1 adds a layer on top of it and must NOT change the v2 scoring logic,
weights, or indicator set.*

---

## 1. Objective

The v2 ranking is anchored to 2023 because the slowest inputs (IRS migration, finalized income)
publish ~2 years late. But most indicators are available far fresher: ZORI is current to within
weeks, permits and BLS employment publish monthly, and Census population *estimates* precede the
finalized detail by a year or more.

v2.1 builds a **nowcast layer**: a provisional current-year score computed from fast-publishing
proxies for the slow indicators, so the framework can produce a provisional 2025 ranking (a
2025→2028 call) *today* instead of in two years.

Non-negotiable principles:
1. **Validate before trusting.** The nowcast is only published if a pseudo-nowcast backtest shows
   it retains most of the full model's accuracy (see M3 and the go/no-go gate).
2. **Provisional is labeled provisional.** Every nowcast run is flagged in the registry and
   reconciled against the finalized score when real data lands. The reconciliation record is part
   of the auditable track record.
3. **This is nowcasting the inputs, not extending the forecast horizon.** The target stays 3-year
   forward rent growth. We do not predict further out than validated.

## 2. The proxy map

For each v2 indicator, classify its publication lag and (if slow) define the fast proxy.
Verify exact current availability at build time — the table reflects expectations, not guarantees.

| v2 indicator | Finalized source (lag) | Fast proxy (lag) | Notes |
| --- | --- | --- | --- |
| net_migration | IRS SOI county flows (~2y) | Census Population Estimates net domestic migration component (~6mo) | THE critical proxy — see M1. Census PEP publishes county-level components of change (births, deaths, net domestic migration) annually, far ahead of IRS detail. |
| job_growth | BLS QCEW (~6-9mo final) | BLS CES/SAE monthly metro employment (~2mo) | Straightforward; CES is timely but noisier and revised. |
| income_growth | BEA regional (~1.5y) | BLS QCEW average weekly wage (~6-9mo) or CES avg hourly earnings | Wage growth as income-growth proxy. |
| permits_to_stock | Census BPS annual + ACS stock (~1y) | Census BPS **monthly** YTD permits, annualized; stock carried forward + prior-year permits | Permits are nearly live; stock moves slowly so carry-forward is safe. |
| rent_to_income | ZORI + BEA income (~1.5y via income) | Current ZORI + wage-growth-projected income | Rent side is live; project the income denominator one year using the wage proxy. |
| cost_to_own_vs_rent | ZHVI + ZORI + FRED mortgage (~weeks) | Same — already fast | No proxy needed. |
| trailing_rent_growth | ZORI (~weeks) | Same — already fast | No proxy needed. |
| employment_diversity | QCEW industry mix (~6-9mo) | Carry forward prior year | Diversification (HHI) is highly persistent year to year; carry-forward is a defensible proxy. Verify persistence empirically in M1. |

Design rule: every proxy must itself be free, public, and reproducible — same standard as v2.

## 4. Milestones

### M1 — Proxy ingestion + the migration-proxy test (the linchpin)

Build `census_pep.py` and `bls_ces.py`; wire proxies into a draft `proxy_map.py`.

Then run the single most important test of the whole spec, because v2's ablation showed
`net_migration` is the one indispensable indicator (dropping it reliably costs τ ≈ 0.15) and it is
also the slowest input: **how well does Census PEP net domestic migration proxy IRS migration?**

- For every metro-year where both exist (~2015-2023), compute the cross-sectional correlation and
  rank correlation between PEP net domestic migration and IRS net migration, per year.
- Also test: does substituting PEP for IRS in the composite change the ranking materially?
  (Spearman between the two resulting rankings, per year.)
- **Interpretation guide:** if the per-year rank correlation is high (roughly ≥ 0.9), the linchpin
  holds and the nowcast is promising. If it is mediocre (< 0.8), flag prominently — the nowcast's
  ceiling is set here, and M3's result will likely be poor. Report whatever it is honestly.

Also in M1: empirically verify the carry-forward proxies (employment_diversity year-over-year rank
correlation; housing-stock persistence).

**Deliverable:** a short auto-generated section in `nowcast-validation.md` with the proxy-quality
tables.

### M2 — The nowcast panel builder

`build_nowcast_panel.py` assembles, for the most recent scoreable year (target: 2025), one row per
metro using the proxy map: fast indicators from live sources, slow indicators from proxies,
carry-forwards where defined.

- Every value carries a provenance tag: `finalized` | `proxy` | `carried_forward`.
- Missing-after-proxy values follow the existing v2 rule (neutral z of 0 at scoring) and are logged.
- Output lands in `data/processed/nowcast/`, never mixed with the finalized panel.
- Reuse the v2 scoring path unchanged to produce a **provisional 2025 ranking**.

**Deliverable:** provisional 2025 cross-section with per-bucket contributions, clearly labeled
provisional, plus a per-metro provenance summary (what fraction of each score rests on proxies).

### M3 — The pseudo-nowcast backtest (the honesty test)

The question: **how much accuracy do the proxies cost?** Answer it on history, where the truth is
known.

- For each historical scoring year (2016-2022 windows, same as the v2 backtest), rebuild that
  year's panel using ONLY what the nowcast would have had: proxies for slow indicators,
  carry-forwards, no finalized IRS/BEA data.
- Run the standard walk-forward evaluation (same metrics: top-weighted τ, precision@10; same
  regimes; same winsorization) on these pseudo-nowcast rankings.
- Report side by side: full-data model vs. pseudo-nowcast, per regime, with bootstrap CIs on the
  gap (reuse the P3 metro-cluster bootstrap machinery).
- Also report ranking agreement: per-year Spearman between the pseudo-nowcast ranking and the
  finalized-data ranking, and top-10 overlap.

**Go/no-go gate (decide the thresholds before running, and log them in the decision log):**
suggested defaults — publish the nowcast if the pseudo-nowcast retains ≥ 85% of the full model's
pooled 3-yr τ AND its top-10 overlap with the finalized ranking averages ≥ 7/10. If it fails,
v2.1 ships as an internal experiment with a documented negative result — which is still a finding
worth a paragraph in the paper.

**Deliverable:** the completed `nowcast-validation.md`, in the same auto-generated style as
`v2-findings.md`.

### M4 — Registry: provisional flag + reconciliation

Extend `registry.py`:

- Frozen runs gain fields: `run_type: finalized | provisional_nowcast`, `proxy_map_version`,
  and the provenance summary.
- New reconciliation record: when finalized data for a nowcast year later arrives, freeze a
  reconciliation entry — provisional ranking vs. finalized ranking, Spearman, top-10 overlap, and
  the per-indicator proxy errors. Never edit the original provisional run.
- The reconciliation history is itself part of the auditable track record (and a future paper
  section: "how good were our provisional calls?").

### M5 — Website integration

- The site's default view may show the provisional ranking ONLY if M3 passed the gate — and then
  always with a visible "Provisional — based on preliminary data" badge, a plain-language
  explainer of what's proxied, and a toggle to the latest finalized ranking (2023).
- Metro drill-down shows provenance per indicator (finalized / proxy / carried forward).
- The track-record page gains a reconciliation section as entries accumulate.

### M6 — Documentation & paper updates

- Append the nowcast decisions to `decision-log.md` (proxy map, gate thresholds, M1/M3 outcomes).
- Add a paper subsection under the data-timing section (6.3 in the current draft): the nowcast
  method, its measured accuracy cost from M3, and the provisional/finalized reconciliation design.
  Frame precisely: *the nowcast shortens the data lag; it does not extend the forecast horizon.*

---

## 5. Guardrails

- **Do not extend the forecast horizon.** 3-year forward rent growth remains the target. No
  5-year claims — zero clean validation windows exist.
- **Do not let proxies leak into the finalized panel** or retrain/reweight anything on proxy data.
  The v2 model is frozen; the nowcast only changes what feeds it.
- **Decide the go/no-go thresholds before M3 runs** — choosing them after seeing results is the
  same sin as picking the metric after the backtest.
- **Label provisional everywhere** — registry, site, paper. A reader should never mistake a
  nowcast ranking for a finalized one.
- **If M1 shows the migration proxy is weak, say so loudly.** The nowcast's value rests on it;
  a weak linchpin honestly reported is worth more than a strong one assumed.

---

## 6. What done looks like

1. A provisional 2025 ranking (2025→2028 call) exists, frozen in the registry as provisional.
2. `nowcast-validation.md` quantifies exactly what the proxies cost, with CIs.
3. The gate decision (publish or internal-only) is logged with its pre-committed thresholds.
4. The site shows provisional vs. finalized transparently (if gated in).
5. The paper's data-timing section explains the layer in plain language.
